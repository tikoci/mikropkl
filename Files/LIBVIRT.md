# Libvirt / QEMU Support

> [!WARNING]
> **Experimental.** Libvirt XML support is generated automatically for all QEMU-backend
> machines and validated against the libvirt RelaxNG schema.  Running under `virsh` on
> Linux CI is supported and tested (see GitHub Actions workflow below).  Running on macOS
> via Homebrew libvirt is **not** a supported configuration.

## Overview

For every `.pkl` manifest with `backend = "QEMU"`, the build produces a `libvirt.xml` file
alongside the UTM bundle in the `Machines/` directory:

```
Machines/
  chr.x86_64.qemu.7.22.utm/
    config.plist        ← UTM configuration
    libvirt.xml         ← libvirt domain XML (generated by Libvirt.pkl)
    Data/
      chr-7.22.img      ← raw RouterOS CHR disk (downloaded by make)
```

The disk paths in `libvirt.xml` use the sentinel `/LIBVIRT_DATA_PATH/` after `pkl eval`
runs.  `make libvirt-fixpaths` (called automatically in `phase2`) replaces this sentinel
with the absolute path to each machine's `Data/` directory.

## UTM → libvirt attribute mapping

| UTM / pkl attribute | libvirt XML element |
|---|---|
| `architecture = "x86_64"` | `<os><type arch="x86_64" machine="q35">` |
| `architecture = "aarch64"` | `<os firmware="efi"><type arch="aarch64" machine="virt">` |
| `memory`  (1 GiB) | `<memory unit="MiB">1024</memory>` |
| `cpus` | `<vcpu placement="static">N</vcpu>` |
| primary disk image | `<disk type="file"><driver type="raw"/>…</disk>` |
| `additionalDisks` (ROSE) | additional `<disk type="file"><driver type="qcow2"/>…</disk>` |
| `Randomish.macAddress(name,…)` | `<interface><mac address="…"/></interface>` |
| QEMU backend | `<domain type="qemu">` |

Key layout decisions:
- **x86_64**: `q35` machine type, ISA serial (`isa-serial`/`isa-serial`), no UEFI by default.
- **aarch64**: `virt` machine type, `firmware="efi"` (libvirt auto-selects UEFI),
  PL011 UART serial (`system-serial`/`pl011`).
- CPU: `host-passthrough` with KVM.  Schema-valid without `migratable` attribute.
- Network: `<interface type="user">` (SLIRP / user-mode NAT, no port forwarding in XML —
  forwarding is added at boot time via `-netdev … hostfwd=` when running directly with QEMU).

## GitHub Actions: Automated Boot Test

The workflow [`.github/workflows/libvirt-test.yaml`](../.github/workflows/libvirt-test.yaml)
builds every QEMU manifest and boots each machine on the matching architecture runner:

| Machine architecture | GitHub runner |
|---|---|
| `aarch64` | `ubuntu-24.04-arm` (native ARM64) |
| `x86_64`  | `ubuntu-latest` (x86_64) |

### What it does

1. **build** job (`ubuntu-latest`): runs `make CHR_VERSION=<version>`, which runs pkl,
   downloads disk images, fixes up libvirt paths, then uploads `./Machines/` as an artifact.
2. **test** job (matrix per machine): downloads the artifact, installs plain QEMU (no
   full libvirt daemon needed), enables KVM via udev rules, extracts VM parameters from
   `libvirt.xml` using `xmllint`, launches QEMU directly with a host-port forward
   (`hostfwd=tcp::9180-:80`), polls `http://localhost:9180/` until RouterOS HTTP is up,
   then calls `/rest/system/check-installation` to verify the installation is healthy.

### Why QEMU is launched directly (not via `virsh start`)

`libvirt.xml` uses `<interface type="user">` for networking, which does not support
`hostfwd` port-forwarding via the libvirt XML alone (that would require a `<qemu:commandline>`
extension).  The workflow launches QEMU directly with the exact flags extracted from the XML,
adding `-netdev user,id=net0,hostfwd=tcp::9180-:80` for the REST API port.

### aarch64 UEFI requirement

RouterOS CHR ARM64 requires UEFI firmware.  The workflow installs the `qemu-efi-aarch64`
Ubuntu package which provides `/usr/share/qemu-efi-aarch64/QEMU_EFI.fd`, passed to QEMU
via `-bios`.

### Disk path handling

`pkl eval` writes `/LIBVIRT_DATA_PATH/<imagename>` in every disk `<source file="">`.
`make libvirt-fixpaths` (part of `phase2`) replaces this sentinel with the absolute path
to the `Data/` directory.  Because GitHub Actions always uses a consistent workspace path
(`/home/runner/work/<owner>/<repo>`), the absolute paths written in the build job are
valid in the test jobs when they download the same artifact.

## Makefile targets

| Target | Description |
|---|---|
| `make libvirt-list` | Print all `libvirt.xml` paths found in `Machines/` |
| `make libvirt-fixpaths` | Replace `/LIBVIRT_DATA_PATH` with real absolute paths (auto-runs in `phase2`) |
| `make libvirt-validate` | Validate all `libvirt.xml` files against the libvirt schema (requires `virt-xml-validate`) |
| `make libvirt-define` | Register all machines with a running libvirtd (`virsh define`) |
| `make libvirt-start` | Start all defined libvirt VMs (`virsh start`) |
| `make libvirt-stop` | Forcibly stop all running VMs (`virsh destroy`) |
| `make libvirt-undefine` | Unregister all VMs from libvirtd (`virsh undefine`) |

## Schema validation (RelaxNG)

After `make` and `make libvirt-fixpaths`:

```sh
# Install libvirt tools
# macOS:
brew install libvirt

# Ubuntu:
sudo apt-get install libvirt-clients libxml2-utils

make libvirt-validate
```

All four QEMU machines (`chr.aarch64`, `chr.x86_64`, `rose.chr.aarch64`, `rose.chr.x86_64`)
validate against the libvirt 12.x RelaxNG schema.

### Known schema constraints encoded in Libvirt.pkl

- `<source file="">` **must** be an absolute path — the schema's `absFilePath` pattern is
  `(/|[a-zA-Z]:\\).+`.  Relative paths like `./Data/…` fail immediately with a misleading
  "Extra element devices in interleave" error.  This is why the `/LIBVIRT_DATA_PATH/`
  sentinel exists.
- `migratable="on"` on `<cpu mode="host-passthrough">` is rejected by some QEMU builds
  (Homebrew QEMU on macOS) and has been removed from the generated XML.
- aarch64 serial must be `type="system-serial"` + `<model name="pl011"/>`.
  x86_64 must be `type="isa-serial"` + `<model name="isa-serial"/>`.

## macOS note

Libvirt/virsh on macOS (Homebrew) has multiple limitations:

- The `<cpu mode="host-passthrough">` requires a KVM-capable QEMU; Homebrew QEMU on macOS
  uses HVKF for ARM guests only, not for x86-on-ARM.
- Emulator path in libvirt.xml is `/usr/bin/qemu-system-*` (Linux default).  Homebrew
  installs to `/usr/local/bin/` or `/opt/homebrew/bin/`.
- `libvirt-fixpaths` writes the macOS absolute path which will not match a Linux deployment.

**For macOS users, use UTM directly** — that is the primary purpose of this project.
Libvirt support targets Linux CI/servers only.
